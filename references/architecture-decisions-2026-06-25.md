# 架构决策记录（2026-06-25）

## 最终架构：3 Cron + Kanban 看板链

### 架构概览

```
Cron × 3                 Kanban（Gateway Dispatcher 自动流转）
┌─────────────┐          ┌────────────────────────────┐
│ 交易员       │          │ 晨间主任务 → 信息员 → 策略员│
│ 每10分钟     │ 读策略   │ （一次性任务链，自动完成）  │
│ 09:15~14:50  │ ──────▶ │                            │
│              │          └────────────────────────────┘
│ 巡检员       │
│ 每30分钟     │
│ 09:25~14:25  │
│              │
│ 策略员盘中   │
│ 12:30/14:30  │
│              │
└─────────────┘

```

### 通信方式
- 所有角色通过 Kanban 评论传递信息（[INFO]/[STRATEGY]/[TRADE]/[PATROL]标签）
- 策略员盘前走 Kanban 自动流转（信息员→完成后自动解锁策略员）
- 交易员和巡检员由 cron 按时触发，读 Kanban 策略、写 Kanban 结果
- 策略员盘中（12:30+14:30）由 cron 触发，读 Kanban 中的[TRADE]+[PATROL]评估

### 各角色职责
| 角色 | 触发方式 | 职责 | 输出 |
|------|---------|------|------|
| 信息员 | Kanban dispatch | 采集外盘/板块/资金/新闻 | Kanban [INFO] |
| 策略员-盘前 | 信息员完成后自动解锁 | 出盘前策略，推送给用户 | Kanban [STRATEGY] + 推送 |
| 交易员 | Cron 每10分钟 | 纯执行策略员指令 | Kanban [TRADE] + 有交易推送 |
| 巡检员 | Cron 每30分钟 | 巡视异动，写报告 | Kanban [PATROL] + 有异动推送 |
| 策略员-盘中 | Cron 12:30/14:30 | 读全天信息判断是否调策略 | Kanban [STRATEGY] + 有变动推送 |

### 关键技术局限（决定当前架构的原因）
1. Kanban 不支持"写评论触发下游"机制
2. Kanban 不支持定时调度
3. Kanban `schedule` 不能调度已完成（done）的任务
4. cron prompt 有长度限制（~200字符内安全）
5. Gateway dispatcher 每60秒扫一次，不支持更细粒度

### 被否决的方案
| 方案 | 否决原因 |
|------|---------|
| 纯看板（方案B） | 无法实现循环任务（交易员每10分钟） |
| 事件驱动触策略 | Kanban 没这能力 |
| 交易巡检一体 cron | 违反"各行其职"原则 |
| 5+个独立 cron job | 太多，管理麻烦 |
